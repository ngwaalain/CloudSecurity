Logical flow — enhanced

All steps with owners and SLAs

1
CloudSec + Arch. Security — org control / rule review Cloud Security + Arch. Security
One-time framework gate. Both teams jointly review all CSPM controls to confirm they are valid, necessary, and appropriate. Controls identified as inapplicable or duplicative are removed from the framework before the exception process begins.
⏱ Periodic review — not per-exception
2
Platform / App team — control review Platform / App team
Sequential step after pre-assessment. Platform and app teams review controls against their resource and platform context. Non-compliant resources or findings are identified here, triggering the choice of remediation path.
⏱ SLA: 2 business days
!
Non-compliant resource / finding identified — choose remediation path
A violation is confirmed. Select the appropriate approach based on where the violation was detected.
Build exception
Caught at IaC / CI/CD build stage via Wiz CI/CD scanner with IaC prevent policy configured in Wiz. Exception applied before resource is deployed.
Runtime exception
Detected in a deployed or live resource via Wiz findings. Resource is already running — exception applied to the live environment.
3
App owner submits proposed exception App owner
Both paths converge here. Owner submits: resource details, business justification, and deviation type — Exception (time-limited), Design, False Positive, or Risk Accepted.
⏱ Owner has 2 business days to submit after finding
4
Platform control review Cloud Security
Cloud Security validates the technical context of the exception request. Confirms resource details are accurate and the control applies.
⏱ SLA: 2 business days
5
Finding review Arch. Security
Arch. Security reviews the finding or control exception in context of the submitted deviation type. Confirms scope and validates justification before risk scoring.
⏱ SLA: 2 business days
6
CargillNow exception ticket created System
Exception ticket opened in CargillNow / ServiceNow. Deviation type recorded and routes to the appropriate approval workflow and weight.
⏱ Same day as step 5 completion
7
Risk assessment Arch. Security + Cloud Security
Both teams jointly score five dimensions: Exposure (0–25), Data sensitivity (0–25), Identity blast radius (0–20), Attack path length (0–15), Criticality (0–15). Score gates the approval path.
⏱ SLA: 3 business days
8a
Approved → notify + implement Cloud Security
ServiceNow auto-notifies owner (1 hr). Cloud Security implements via Wiz ignore policy or IaC PR gate — 1 reviewer, Wiz scan must pass, linked to CargillNow ticket, expiry date set in IaC metadata.
⏱ Notify: 1 hr automated · Implement: 2 business days
9
Expiry enforcement ServiceNow automated
T-30 warning to owner. T-7 manager escalation if no action. T+0 Wiz ignore policy auto-revoked, resource re-enters findings queue, ticket closed as expired.
⏱ Fully automated
8b
Rejected → remediation loop Cloud Security / Arch. Security
Rejection recorded with reason code. Owner options: fix resource, re-submit with stronger justification, or escalate to Manager. No response by day 5 triggers manager auto-notification and non-compliant flag in Wiz.
⏱ Owner SLA: 5 business days

Deviation type — approval routing

False Positive
Lightweight path — control flagged something that is not a real risk. Technical validation only, no risk committee needed.
Platform engineer (confirms FP)Cloud Security sign-off
⏱ Total SLA: 2 business days
Design exception
Architectural review required — control conflicts with an intentional design pattern. May recur across resources so the pattern is assessed holistically.
Platform control reviewArch. SecurityApp owner sign-off
⏱ Total SLA: 5 business days
Exception (time-limited)
Standard path with mandatory expiry. Temporary deviation acceptable with justification. Renewal requires full re-justification — not auto-renewed. Max 90 days.

⏱ Total SLA: 5 business days · Max: 90 days

Heaviest path — permanent or long-term acceptance of known risk. Requires Manager sign-off and mandatory annual review. Score ≥70 escalates to Manager regardless.

Rejection and remediation flow

R0
Approval denied Arch. Security / Cloud Security
Risk assessment results in denial. Recorded in CargillNow with team, risk score, and specific reason.
⏱ Automated within 1 hour
R1
Rejection issued with guidance Cloud Security
CargillNow updated with reason code, control violated, and three options: (1) fix the misconfiguration, (2) re-submit with stronger justification, (3) escalate to Manager for risk acceptance.
⏱ Owner must respond within 5 business days
R2
Owner response App owner
Fix: Wiz scanner auto-validates. Re-submit: re-enters flow at step 3 with rejection history pre-populated. Escalate: Manager review added to standard risk path.
⏱ Deadline: day 5
R3
No response — escalation Automated
Manager added to ticket. Resource tagged non-compliant in Wiz. Finding escalated to Manager dashboard. Stays non-compliant until resolved.
⏱ Auto-triggered at day 5

Auto-expiry lifecycle — ServiceNow managed

Day 0 — exception approved and implemented
Expiry date set in CargillNow. Wiz ignore policy applied with expiry metadata. ServiceNow auto-task created — no manual reminders needed.
T-30 days — first warning
ServiceNow notifies resource owner and app owner. Owner prompted to initiate renewal (re-enters flow at step 3) or close if no longer needed.
T-7 days — escalation if no renewal action
Manager auto-added to CargillNow ticket. Email sent as secondary. Exception flagged on Manager dashboard as "expiring — no action."
T+0 — auto-revocation
Wiz ignore policy automatically removed. Resource re-enters findings queue. Ticket closed as "expired — not renewed." IaC metadata expiry hook triggers removal.
Renewal path (any time before T+0)
Owner triggers renewal from CargillNow. Re-enters workflow at step 3 with prior history pre-populated. Risk score re-evaluated.

GitHub Issues flow — current temporary intake

⏳ Temporary process — GitHub Issues intake
This flow uses GitHub Issues as the intake mechanism instead of CargillNow. All approval, implementation, risk scoring, and expiry steps are identical to the future-state flow. Once CargillNow is ready, submission and notifications move there — everything else stays the same.
1
Framework pre-assessment Cloud Security + Arch. Security
One-time gate. Both teams jointly review all CSPM controls to confirm they are valid and necessary. Controls identified as inapplicable are removed before the exception process begins.
⏱ Periodic — not per exception
2
Finding identified Platform / App team
Platform and app teams review controls against their resource context. Non-compliant findings are identified — either at build stage (Wiz CI/CD scanner) or runtime (Wiz findings). Both paths converge at step 3.
Build exception
Caught at IaC / CI-CD stage before resource is deployed. Lower urgency.
Runtime exception
Detected in a live, deployed resource via Wiz findings. Higher urgency.
⏱ SLA: 2 business days to identify and classify
3
Submit via GitHub Issue ⏳ App owner
App owner creates a GitHub Issue at cs-team-collaborations/issues with all required fields: resource details, AWS account, Wiz rule ID and UUID, deviation type, business justification, risk description, compensating controls, requested expiry date, and ITSM ticket reference (e.g. ADC-123). Exception Terraform code is auto-generated from the issue on approval.
⏱ Owner has 2 business days to submit after finding
4
Platform control review Cloud Security
Cloud Security validates the technical context of the exception request. Confirms resource details are accurate and the control applies as stated.
⏱ SLA: 2 business days
5
Finding review Arch. Security
Arch. Security reviews the finding in context of the submitted deviation type. Validates justification and scope before risk scoring.
⏱ SLA: 2 business days
6
Risk assessment Arch. Security + Cloud Security
Both teams jointly score five dimensions: Exposure (0–25), Data sensitivity (0–25), Identity blast radius (0–20), Attack path length (0–15), Criticality (0–15). Score gates the approval path: 0–29 expedited · 30–69 standard · ≥70 Manager escalation.
⏱ SLA: 3 business days
7a
Approved — notify + implement Cloud Security
Approval comment added to the GitHub Issue. Cloud Security implements via Wiz ignore policy or IaC PR gate. Branch naming: exception/GH-{issue-id}-{deviation-type}. PR linked to GitHub Issue. Wiz CI/CD scan must pass. Minimum 1 reviewer. Expiry date set in IaC metadata.
⏱ Notification: same day · Implementation: 2 business days
8
Expiry enforcement ⏳ Vela pipeline + email
T-30: email notification to resource owner and app owner. Owner prompted to initiate renewal or close if no longer needed.
T-7: manager auto-notified via email if no renewal action taken. Exception flagged as "expiring — no action."
T+0: Wiz ignore policy automatically removed via Vela pipeline. Resource re-enters findings queue. GitHub Issue updated as expired.
⏱ Fully automated via Vela pipeline
7b
Rejected — remediation loop Cloud Security / Arch. Security
Rejection comment added to the GitHub Issue with reason and three options: (1) fix the misconfiguration and close, (2) resubmit with stronger justification — re-enters flow at step 3, (3) escalate to Manager for risk acceptance review.

No response within 5 business days: Manager auto-notified via email. Resource flagged as non-compliant in Wiz until resolved.
⏱ Owner SLA: 5 business days to respond
